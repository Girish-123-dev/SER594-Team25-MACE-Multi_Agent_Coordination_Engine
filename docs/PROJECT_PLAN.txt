1. PROJECT OVERVIEW
================================================================================

What MACE Does:
  MACE is a multi-agent orchestration system. A user types a natural-language
  request into a web UI. The system:
    1. Parses the user's intent using an LLM (Claude)
    2. Routes subtasks to the right agent (Support Agent or Domain Agent)
    3. Detects conflicts (duplicate intents, resource clashes, dependency chains)
    4. Resolves conflicts via rule-based or LLM-powered arbitration
    5. Stores everything in shared memory so agents coordinate, not collide

Who It's For:
  Teams or orgs running multiple AI agents that need to work together without
  duplicating work, conflicting on resources, or breaking sequential workflows.

Why It Matters:
  Current multi-agent setups are isolated silos. Each agent has no awareness of
  what others are doing. MACE adds the coordination layer that's missing.

9. AI TECHNIQUES — DEEP BREAKDOWN
================================================================================

  We implement 4 techniques. The spec says 3+, but 4 gives us depth.

  ─────────────────────────────────────────────────────────────────────────
  TECHNIQUE 1: LLM API INTEGRATION (Anthropic Claude)
  ─────────────────────────────────────────────────────────────────────────
  File: backend/app/services/llm.py

  What it does:
    Calls Claude to parse user intent, generate arbitration decisions,
    and produce structured JSON outputs.

  Spec requirements (§4.1):
    ✓ API key management       → loads from .env via Settings
    ✓ Error handling + retries → 3 retries with exponential backoff
    ✓ Token/cost tracking      → log input_tokens, output_tokens per call
    ✓ Swappable wrapper → abstract LLMService class; swap Claude for
                                  OpenAI or local model by changing one class

  Code structure:
    class LLMResponse(BaseModel):
        content: str
        parsed: dict | None
        input_tokens: int
        output_tokens: int
        model: str
        latency_ms: float

    class LLMService(ABC):
        async def complete(prompt, system_prompt, output_schema) -> LLMResponse

    class AnthropicService(LLMService):
        # Concrete implementation using anthropic SDK
        # Retry logic, token counting, structured output parsing

  How we evaluate it:
    • Token cost per request (track across 50 test inputs)
    • Error rate (% of failed API calls after retries)
    • Latency p50/p95

  ─────────────────────────────────────────────────────────────────────────
  TECHNIQUE 2: AI AGENTS / MULTI-STEP WORKFLOWS (LangGraph)
  ─────────────────────────────────────────────────────────────────────────
  File: backend/app/orchestrator/graph.py

  What it does:
    The Orchestrator is a LangGraph state graph that decomposes user input
    into subtasks, routes to agents, handles conflicts, and produces a
    final coordinated response. It's NOT a single LLM call — it's a
    multi-node graph with conditional edges.

  Graph nodes:
    1. parse_intent     → Claude extracts intent type, entities, priority
    2. check_duplicate  → FAISS search for similar pending intents
    3. route_task       → Decide: Support Agent or Domain Agent (or both)
    4. detect_conflict  → Check resource locks + dependencies
    5. arbitrate        → If conflict, resolve via rules or Claude
    6. execute_agent    → Call the actual agent function
    7. update_memory    → Write results to shared memory

  Conditional edges:
    • check_duplicate: if duplicate found → skip to END (dedup)
    • detect_conflict: if no conflict → skip arbitrate → execute_agent
    • route_task: if both agents needed → parallel branch (or sequential)

  Tools (spec requires 3+):
    Tool 1: create_ticket(summary, priority)
    Tool 2: query_faq(question)
    Tool 3: escalate_to_human(ticket_id, reason)
    Tool 4: domain_lookup(entity_id)

  How we evaluate it:
    • Task completion rate on 30 curated scenarios
    • Average steps to completion
    • Tool selection accuracy (correct agent + tool chosen)

  ─────────────────────────────────────────────────────────────────────────
  TECHNIQUE 3: VECTOR SEARCH / EMBEDDINGS (FAISS)
  ─────────────────────────────────────────────────────────────────────────
  Files: backend/app/memory/embeddings.py, faiss_store.py

  What it does:
    Embeds every user intent into a vector, stores in FAISS index,
    and searches for similar intents to detect duplicates.

  Pipeline:
    1. User submits "reset my password"
    2. sentence-transformers (all-MiniLM-L6-v2) → 384-dim vector
    3. Search FAISS index for vectors with cosine similarity > 0.85
    4. If match found → flag as duplicate intent
    5. Add new vector to index
    6. Persist index to disk (FAISS_INDEX_PATH)

  NOT a single API call — it's a full pipeline:
    • Custom embedding logic (not just calling an API)
    • Index management (add, search, persist, rebuild)
    • Threshold tuning (SIMILARITY_THRESHOLD in .env)
    • Integration with conflict detection

  How we evaluate it:
    • Precision@5 for duplicate detection
    • Recall@5 for duplicate detection
    • Baseline: exact string match vs. FAISS semantic match

  ─────────────────────────────────────────────────────────────────────────
  TECHNIQUE 4: MEMORY / CONVERSATION MANAGEMENT
  ─────────────────────────────────────────────────────────────────────────
  File: backend/app/memory/shared.py

  What it does:
    Maintains persistent shared state across all agents and orchestration
    cycles. Agents read and write to shared memory so they're aware of
    each other's actions.

  Components:
    a) Task Registry (SQLite)
       - All tasks: id, user_id, intent, status, assigned_agent, result
       - Agents check this before acting to avoid conflicts

    b) Agent Status (in-memory + SQLite)
       - Which agent is busy, idle, last action timestamp
       - Used by router for load balancing

    c) Conflict Log (SQLite)
       - Every conflict: type, involved tasks, resolution, timestamp
       - Shown in the UI for transparency

    d) Intent Store (FAISS)
       - All embedded intents for deduplication

    e) Conversation Context (SQLite)
       - Per-user rolling window of last N interactions
       - If window exceeds token limit → summarize via Claude
       - User preferences learned over time

  How we evaluate it:
    • Context retention accuracy: ask follow-up questions, check if
      system remembers prior context
    • Memory retrieval relevance: score on 20 curated multi-turn
      conversations

10. THREE DEMO SCENARIOS — EXACTLY HOW THEY WORK
================================================================================

  These are the 3 scenarios we show in the demo video to prove coordination.

  ─── SCENARIO 1: DUPLICATE INTENT ───────────────────────────────────────

  Setup:
    User A sends: "I forgot my password, please help"
    30 seconds later, User A sends: "Can you reset my password?"

  What happens:
    1. First message → parse_intent → "password_reset" intent
    2. Embed intent → add to FAISS index → route to Support Agent
    3. Support Agent creates ticket #101 → saved to shared memory
    4. Second message → parse_intent → "password_reset" intent
    5. Embed intent → FAISS search → similarity 0.93 with ticket #101
    6. Conflict Detector flags: DUPLICATE_INTENT
    7. Resolution: Merge — return existing ticket #101 to user
    8. UI shows: "This looks similar to your existing request (Ticket #101).
       We're already working on it."

  What this proves:
    FAISS-based semantic deduplication prevents wasted work.

  ─── SCENARIO 2: CONFLICTING UPDATE ─────────────────────────────────────

  Setup:
    User sends: "Update my laptop's status to 'returned' and also
    create a support ticket about the laptop screen issue"

  What happens:
    1. parse_intent → TWO subtasks:
       a) Domain Agent: update_asset_status(laptop_123, "returned")
       b) Support Agent: create_ticket("laptop screen issue", high)
    2. route_task → both agents assigned
    3. detect_conflict → RESOURCE_CLASH:
       Domain Agent wants to mark laptop as "returned"
       Support Agent wants to create active ticket for same laptop
       (returned laptop shouldn't have active issues)
    4. Arbitration Engine → Claude decides:
       "The laptop should remain 'active' until the screen issue is
        resolved. Create the support ticket first, then update status
        after resolution."
    5. Resolution: Reorder tasks — ticket first, status update deferred
    6. UI shows conflict card with both options + AI recommendation

  What this proves:
    The system detects when agents would create an inconsistent state
    and uses LLM arbitration to resolve it.

  ─── SCENARIO 3: SEQUENTIAL DEPENDENCY ──────────────────────────────────

  Setup:
    User sends: "Diagnose why my VPN isn't connecting, and once you
    figure it out, create a ticket with the diagnosis"

  What happens:
    1. parse_intent → detects sequential dependency ("once you figure it out")
    2. Two subtasks with dependency:
       Task A: Domain Agent → diagnose_vpn(user_id) [MUST complete first]
       Task B: Support Agent → create_ticket(diagnosis) [DEPENDS on A]
    3. detect_conflict → SEQUENTIAL_DEPENDENCY (B depends on A's output)
    4. Orchestrator queues Task B, executes Task A first
    5. Domain Agent returns: "VPN certificate expired on 2026-04-01"
    6. Orchestrator feeds diagnosis into Task B
    7. Support Agent creates ticket: "VPN connectivity failure —
       certificate expired on 2026-04-01. Renewal required."
    8. UI shows the chain: Task A → output → Task B → final result

  What this proves:
    The orchestrator can chain agent outputs where one feeds into the next.

12. DATABASE SCHEMA
================================================================================

  SQLite (backend/data/mace.db):

  TABLE users
    id              INTEGER PRIMARY KEY AUTOINCREMENT
    username        TEXT UNIQUE NOT NULL
    email           TEXT UNIQUE NOT NULL
    password_hash   TEXT NOT NULL
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  TABLE tasks
    id              INTEGER PRIMARY KEY AUTOINCREMENT
    user_id         INTEGER NOT NULL → users.id
    intent          TEXT NOT NULL
    status          TEXT DEFAULT 'pending'   -- pending | in_progress | completed | failed | duplicate
    assigned_agent  TEXT                     -- 'support' | 'domain' | 'both'
    result          TEXT
    parent_task_id  INTEGER                  -- for sequential dependencies
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  TABLE conflicts
    id              INTEGER PRIMARY KEY AUTOINCREMENT
    task_id_a       INTEGER NOT NULL → tasks.id
    task_id_b       INTEGER NOT NULL → tasks.id
    conflict_type   TEXT NOT NULL            -- 'duplicate_intent' | 'resource_clash' | 'sequential_dependency'
    resolution      TEXT                     -- 'merged' | 'reordered' | 'priority_override' | 'user_override'
    resolved_at     TIMESTAMP
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  TABLE conversations
    id              INTEGER PRIMARY KEY AUTOINCREMENT
    user_id         INTEGER NOT NULL → users.id
    role            TEXT NOT NULL            -- 'user' | 'assistant'
    content         TEXT NOT NULL
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP

  FAISS Index (backend/data/faiss_index/):
    intent_index.faiss   — 384-dimensional vectors
    intent_map.json      — maps FAISS vector ID → task ID

15. EVALUATION PLAN (METRICS + BASELINES)
================================================================================

  ── AI Metrics (spec §5.2 — need at least 2) ──

  Metric 1: Duplicate Intent Detection
    Metric:    Precision@5 and Recall@5

    Baseline:  Exact string match
    Expected:  FAISS (P@5 > 0.85) vs. string match (P@5 ~ 0.40)

  Metric 2: Task Completion Rate
    Metric:    % of scenarios where orchestrator produces correct final output
    Dataset:   30 curated user scenarios (10 per demo type)
    Baseline:  Direct single-agent call (no orchestration)
    Expected:  Orchestrated (>80%) vs. direct (>50%)

  Metric 3: Tool Selection Accuracy
    Metric:    % of scenarios where correct agent + tool is chosen
    Dataset:   Same 30 scenarios
    Baseline:  Random agent assignment
    Expected:  LangGraph routing (>85%) vs. random (~33%)

  ── System Metrics (spec §5.3) ──

  Metric:           How Measured:
  Latency p50       Time from request to response, median over 50 runs
  Latency p95       95th percentile of same
  Error rate        % of requests returning 5xx over 100 runs
  Test coverage     pytest --cov, report as % of backend code covered

  All results written to: eval/results.json
  Script: eval/run_eval.py
